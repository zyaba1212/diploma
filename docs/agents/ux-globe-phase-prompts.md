# UX / Globe phase — промпты агентов (A–G)

Файл для **координатора** и **агентов**: сначала прочитать **общий промпт** ниже, затем выполнить **только свою букву (A–G)**. Маппинг роль → буква — в конце файла и в `AGENTS.md` (раздел «UX / Globe phase»).

**Связанные документы:** `docs/requirements.md`, `docs/architecture.md`, `docs/network-data-and-sources.md`, `docs/earth-visualization.md`, `AGENTS.md`.

---

## Общий промпт (вставить в чат агенту при старте работ)

Скопируйте блок целиком. **Букву не подставляет координатор** — агент сам находит свою букву по **роли** (таблица «Маппинг буква → роль» ниже в этом файле или в `AGENTS.md` → раздел «UX / Globe phase»).

```
Ты работаешь по файлу docs/agents/ux-globe-phase-prompts.md.

1) Прочитай этот файл целиком (общий контекст + секции A–G).
2) Определи свою букву (A–G) по своей роли из таблицы маппинга в конце файла и выполни ТОЛЬКО эту секцию — не меняй чужие зоны без согласования.
3) Сверяйся с docs/requirements.md и docs/architecture.md; публичные контракты API не ломай без явной пометки breaking change.
4) После изменений: npm run lint && npm run build; зафиксируй итог в DEVELOPMENT_JOURNAL.md (кратко: что сделано, файлы).
```

Координатор при необходимости только добавляет: роль (если неочевидно), контекст задачи, **разрешение на правки** в репозитории.

## A) RefactorGuardianAgent — preflight

- Зафиксировать **инварианты**: маршруты `/api/network`, `/api/proposals/*`, форматы DTO в `src/lib/types.ts` — не ломать без задачи.

- Проверить, что новые UI-элементы не меняют семантику существующих API.

## B) ArchitectAgent — решения по продукту (коротко зафиксировать в docs)

- **2D vs 3D:** авто-переключение по zoom не должно конфликтовать с ручным выбором; переход 3D -> 2D обязан центрировать карту в текущую “середину” глобуса.
- **Читаемость Земли:** осветлить 3D-сцену (фон/тон/экспозиция/эмиссив-fallback) чтобы элементы были различимы.
- **Локальные подписи на 3D:** подписи показывать только в окрестности текущего центра глобуса (и/или при zoom-in), а не “всё сразу”.
- **Подземные кабели vs подводные:** различать submarine и underground кабели цветом/глубиной/уровнем; использовать расширенные типы сети из БД.
- **2D как Google/Yandex-аналоги:** на 2D показывать текущую локацию (страна/область/город) через reverse geocode + поиск поселений с автокомплитом; внешний geocoding всегда через backend proxy.

## C) Web3SolanaAgent — кошелёк Phantom

- Починить **Connect / Disconnect** на главной (`AuthBlock.tsx`, `providers.tsx`): актуальный паттерн `@solana/wallet-adapter-react` (при необходимости `WalletModalProvider` / обработка ошибок / не блокировать UI).
- Убедиться, что `autoConnect` и ручной connect не конфликтуют; статус `connected` отображается корректно.
- **Критерий:** пользователь может подключить и отключить кошелёк в браузере с установленным Phantom.

---

## D) ThreeLeafletAgent — глобус, карта, зум, свет, сеть

- **Зум:** колесо/тачпад не должны «ломать» интерфейс; синхронизировать с состоянием камеры; добавить кнопки **+ / −** (тот же clamp, что у wheel).
- **2D:** клик «2D» стабильно показывает `MapView`; исправить логику `viewMode` / пороги `ZOOM_THRESHOLD_*`, чтобы пользователь не терял режим.
- **Центр 2D из глобуса:** при переключении 3D -> 2D центрировать Leaflet в текущую середину глобуса (там, куда “смотрит” камера / текущий center state).
- **Текущая локация на 2D:** показывать “слева сверху” Country/Region/City по reverse geocode для центра карты (через backend proxy).
- **Слои UI:** не перекрывать панели навигации/авторизации — согласовать **z-index** с `FrontendAgent`/`HomePage` (канвас и оверлей карты ниже шапки).
- **Земля светлее:** фон сцены, интенсивности света, при необходимости `toneMappingExposure` / env из `docs/earth-visualization.md`.
- **Сеть:** различимые материалы/цвета для `CABLE_*`, `BASE_STATION`, `SATELLITE`, `EQUIPMENT`; при перегрузе — фильтр или легенда «что видно».
- **Подписи на 3D:** отрисовывать только “локально” (в радиусе/по текущему центру глобуса) и обновлять при изменении центра/zoom.
- **Кабели в 3D:** опционально позже — raycast/pick; если не делается — не блокировать список/карточки из секции E/F.

## E) FrontendAgent — layout, страницы, панели

- **HomePage / шапка:** гарантировать видимость навигации и `AuthBlock` поверх полноэкранной сцены (`z-index`, `pointer-events` по необходимости).
- **Кнопки зума ±** рядом с отображением zoom (если размещены не в ThreeLeaflet — координация с D).
- **2D UI overlay (аналог Google/Yandex):**
  - слева сверху: текущая локация (Country/Region/City) по reverse geocode;
  - ниже: строка поиска поселения с автокомплитом (сначала показывать 3 результата; если результатов больше — список с прокруткой);
  - по выбору: выполнить recenter карты в выбранную точку.
- **Кабели / информация:** минимум — боковая панель или страница `/cables` (или аналог): таблица/список элементов с типом, ссылка на деталь; деталь — имя, поля из `metadata`.
- Состояния загрузки и ошибок сети.

## F) BackendAPIAgent + ScriptsImportAgent

**BackendAPIAgent**

- При необходимости: `GET /api/network/elements/[id]` или расширение существующего `GET /api/network` (не ломая клиентов) — отдача `metadata`, `provider`, для карточки кабеля.
- Кэш-заголовки / лимиты по существующим правилам проекта.
- **Geocode proxy (через внешний API, но не напрямую из браузера):**
  - `search` (autocomplete): по запросу `q` вернуть топ N кандидатов для списка/автокомплита;
  - `reverse` (центр): по `lat/lng` вернуть Country/Region/City (или эквивалентные поля);
  - `nearby/local labels` для локальных 3D подписей (по текущему центру/радиусу).
  - rate limit + кэш (следовать уже существующим правилам `src/lib/rateLimit`).

**ScriptsImportAgent**

- При расширении карточки кабеля: догрузить в `metadata` поля из open dataset (owners, rfs, length, url) при следующем импорте — **без** нарушения лицензии (CC BY-NC-SA и т.д. — см. `docs/network-data-and-sources.md`).
- Реализовать `scripts/sync-underground-cables.mjs` и расширить `docs/network-data-and-sources.md` актуальной ссылкой на dataset (без заглушек).
- Убедиться, что `metadata` содержит поля, нужные для визуального различия submarine vs underground (цвет/глубина/тип/доп. атрибуты — конкретные поля согласовать с ArchitectAgent).
