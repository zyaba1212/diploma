# Stage 12 — scope (архитекторский черновик)

Цель этапа: **закрыть разрыв между «задокументировано в Stage 11» и «реально работает в проде»**, плюс дать **минимальную управляемость** потока предложений (governance / moderation baseline), не ломая публичные контракты API Stage 5–8.

## 1) Реализация инфраструктурных решений Stage 11 (implementation gap)

Stage 11 описал multi-instance, Redis-ready rate limit, пулы к БД — Stage 12 **внедряет** выбранное минимумом, согласованным с `docs/stage11-post-launch-architecture.md` и `docs/operations.md`:

- **Единый rate limit с разделяемым состоянием** при 2+ инстансах: Redis (или эквивалент) за интерфейсом, совместимым с текущим поведением `429` и заголовками.
- **Connection pooling**: рекомендации из Stage 11 → фактическая конфигурация в целевом окружении (PgBouncer / managed pool), без смены семантики Prisma-запросов.
- **Жёсткие лимиты body** на mutation-маршрутах proposals — единая политика и тесты/документация.

Критерий: поведение API для клиента **не меняется** (те же коды, форматы ошибок), меняется только корректность под нагрузкой и при горизонтальном масштабировании.

## 2) Governance / moderation baseline (продукт + безопасность)

Сейчас цепочка proposals уже есть, но **управляемый поток** (кто переводит SUBMITTED → ACCEPTED/REJECTED, аудит) для «живой» эксплуатации недостаточно явен:

- **Роли**: разделение «автор предложения» vs «модератор/оператор» (хотя бы через allowlist pubkey + env, без полноценной RBAC-системы в v1).
- **Минимальный API и/или защищённый UI** для смены статусов по правилам, с записью в журнал (audit trail), согласованным с `HistoryEntry`/операциями где применимо.
- **Политика**: что считается принятием решения, идемпотентность, защита от двойного применения.

Новые маршруты — **по возможности с префиксом** вроде `/api/moderation/*` или `/api/admin/*`, чтобы не трогать существующие контракты `/api/proposals/*` без необходимости.

## 3) Наблюдаемость и приёмка под нагрузкой (узкий слой)

- Алерты/SLO из `docs/operations.md` привязать к **фактическим** метрикам после Redis/pool.
- Один сценарий **load/smoke** (даже ручной чеклист): N инстансов + общий Redis → согласованный rate limit.

## 4) Что сознательно не входит в Stage 12 (чтобы не размыть этап)

- Полная замена Memo на «боевой» Anchor-программный контракт (это отдельный крупный этап).
- Мультитенантность, мультирегион active-active, мобильные приложения.
- Полноценный маркетинговый лендинг и i18n — можно вынести в Stage 13+.

## Статус по факту реализации (по коду сейчас)

- Shared rate limit state:
  - частично реализовано: `src/lib/rateLimit.ts` поддерживает переключение на Redis через `RATE_LIMIT_BACKEND=redis` и `REDIS_URL`;
  - при недоступном Redis поведение фолбэкается в in-memory режиме;
  - форматы ответов `429 { error: "rate limit exceeded" }` сохраняются (контракт не ломается).
- Connection pooling:
  - пока не имеет зафиксированной “целевой конфигурации” в виде отдельной реализации/документа (Prisma подключается в текущем виде).
- Body size / mutation guards:
  - частично: лимит по `elementPayload` (guard “too large”) реализован только в `POST /api/proposals/:id/actions`;
  - единая политика “на все mutation routes proposals” пока отсутствует.
- Moderation / governance baseline:
  - реализовано: `POST /api/moderation/proposals/:id/decide`;
  - allowlist модераторов задаётся `MODERATOR_PUBKEYS` (comma-separated base58);
  - optional signature-проверка Phantom делается, только если в request body передан `signature`;
  - в moderation endpoint также используется rate limit (ключ `moderation.decide:<clientIp>`);
  - body size guard выполнен через `content-length` и возвращает `400 { error: "payload too large" }` при превышении лимита.
